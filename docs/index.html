Why Flexers?

Built for the Modern Rust Ecosystem

⚡

Zero FFI Overhead

Pure Rust implementation eliminates FFI marshalling costs. Direct memory access and native function calls.

🛡️

Memory Safe

No null pointers, no use-after-free, no data races. Rust's ownership system prevents entire classes of bugs.

🔧

Easy Integration

Standard Cargo crate. No manual bindings, no build scripts for C compilation, no platform-specific hacks.

🐛

Better Debugging

Full backtraces, thread-safe by design, clear error types. No pointer aliasing mysteries.

📦

Modular Design

Clean separation: flexers-core (CPU), flexers-periph (I/O), flexers-stubs (ROM), flexers-session (API).

⚙️

Performance First

Page-table memory lookup, inline fast paths, cache-aligned CPU state, compiler jump table dispatch.

Development Timeline

9-Week Roadmap to LVGL Demo

Phase 1: Core CPU & Memory ✓ 85% Complete

Weeks 1-3 • Xtensa CPU, memory subsystem, 30+ instructions, execution loop

  • ✓ XtensaCpu with windowed registers
  • ✓ Page-table memory (520KB SRAM, 448KB ROM, 4MB flash)
  • ✓ Instruction fetch/decode (16-bit/24-bit)
  • ✓ ALU, load/store, branches, calls, special registers
  • ⏳ Binary loader (ESP32 format)
  • ⏳ Integration tests

Phase 2: Peripherals & I/O

Weeks 4-5 • UART, timers, interrupts for firmware boot

  • MMIO handler dispatcher
  • UART0/1/2 with callbacks
  • CCOUNT/CCOMPARE timer
  • Interrupt delivery (INTENABLE, PS masking)

Phase 3: ROM Stubs & Symbols

Weeks 6-7 • Fast ROM function interception

  • PC hook bitmap (512K bits, O(1) lookup)
  • Direct dispatch table (64K entries)
  • Core stubs: ets_printf, memcpy, strcmp
  • ELF symbol loading (goblin crate)

Phase 4: Display Integration

Week 8 • TFT_eSPI/LVGL framebuffer rendering

  • TFT_eSPI::pushColors hook
  • Framebuffer manager (RGB565)
  • Touch input simulation
  • LVGL demo validation

Phase 5: Cyders Integration

Week 9 • Drop-in replacement for C flexe

  • Session API matching FlexeSession
  • Remove FFI dependencies
  • Performance benchmarking
  • Full LVGL demo support

Phase 6: JIT Compilation

Future • If benchmarking reveals performance gaps

  • Basic block compilation
  • Register allocation
  • Hot path detection

Design Optimizations

Component Optimization Impact
Memory Access Page table + inline fast path O(1) lookup, no bounds checks
CPU State 64-byte cache alignment Hot registers in single cache line
Dispatch Nested match statements Compiler jump tables (vs function pointers)
Integration Zero FFI overhead Direct Rust function calls
Memory Model UnsafeCell for Arc<Memory> No RefCell runtime overhead
